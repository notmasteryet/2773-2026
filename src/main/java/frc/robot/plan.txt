## Potential Challenges & Solutions

### Challenge 1: RobotConfig Parameters Missing
PROBLEM: Physical specs (mass, MOI, wheel radius) not in codebase
SOLUTION: Measure robot or use conservative estimates
        - Mass: 50kg (typical FRC robot)
        - MOI: 6.0 kg·m² (estimate based on wheelbase)
        - Wheel radius: 0.0508m (from circumference 0.1016m diameter)
RISK: Wrong values cause poor path tracking

### Challenge 2: Gyro Sign Convention
PROBLEM: Code shows gyro.getRotation2d().times(-1) in multiple places
SOLUTION: Test with straight path; if robot spins, flip sign
RISK: Robot rotates opposite direction during autonomous

### Challenge 3: ChassisSpeeds Conversion
PROBLEM: chassisDrive() converts to polar coordinates with custom ratios
        PathPlanner expects direct ChassisSpeeds (m/s) input
SOLUTION: May need to rewrite chassisDrive() to use proper
        SwerveDriveKinematics.toSwerveModuleStates()
RISK: Robot doesn't follow paths correctly

### Challenge 4: Module State Optimization Missing
PROBLEM: No desaturation or angle optimization in current drive code
SOLUTION: Add SwerveDriveKinematics.desaturateWheelSpeeds()
        Add module state optimization (minimize rotation)
RISK: Modules spin unnecessarily, speed limits exceeded

### Challenge 5: AutoBuilder Initialization Timing
PROBLEM: Must initialize before any path commands are created
SOLUTION: Initialize in DriveSubsystem constructor or static block
        Ensure DriveSubsystem created before RobotContainer uses AutoBuilder
RISK: Null pointer exceptions

### Challenge 6: Path File Deployment
PROBLEM: Current deploy task copies to /home/lvuser/deploy/
        PathPlanner needs /home/lvuser/pathplanner/paths/
SOLUTION: Update build.gradle deploy task OR
        Manually copy path files to roboRIO OR
        Use PathPlanner's recommended deployment method
RISK: Paths work in simulation but not on robot

### Challenge 7: PID Tuning
PROBLEM: Current PID (P=0.63, I=0, D=0) may not work for path following
SOLUTION: Tune separately for:
        - Translation PID (start P=1.0, I=0, D=0)
        - Rotation PID (start P=0.5, I=0, D=0.1)
        Test and adjust based on path tracking performance
RISK: Oscillation, overshooting, poor tracking

### Challenge 8: Field Coordinate System
PROBLEM: getY() returns -pose.getY() (Y is negated)
SOLUTION: Verify field orientation matches PathPlanner expectations
        Test with known path and check direction
RISK: Paths mirrored across field

### Challenge 9: Module Encoder Offsets
PROBLEM: Current offsets may not be calibrated for PathPlanner
SOLUTION: Run PathPlanner's module verification routine
        Adjust offsets if modules don't point correctly
RISK: Robot drifts, modules fight each other

### Challenge 10: Vision During Autonomous
PROBLEM: Vision updates continue during path following
SOLUTION: Ensure stdDevs properly weight vision reliability
        Consider disabling vision for initial testing
RISK: Position jumps from bad tag detections

### Challenge 11: Simulation Support Disabled
PROBLEM: includeDesktopSupport = false in build.gradle
SOLUTION: Temporarily enable for testing: includeDesktopSupport = true
RISK: Must debug entirely on robot (time-consuming)

## PathPlannerLib API Reference

### AutoBuilder.configure() Parameters
- poseSupplier: Supplier<Pose2d> - Current robot pose
- resetPose: Consumer<Pose2d> - Reset odometry
- robotConfig: RobotConfig - From settings.json
- driveController: PPHolonomicDriveController - PID controllers
- driveConsumer: Consumer<ChassisSpeeds> - Execute speeds
- rotationSupplier: Supplier<Rotation2d> - Target rotation (optional)
- flipSupplier: Supplier<Boolean> - Flip for red alliance (optional)

### Common Commands
- AutoBuilder.followPath(PathPlannerPath) - Follow single path
- AutoBuilder.pathfindToPose(Pose2d, PathConstraints) - Pathfind to pose
- AutoBuilder.pathfindThenFollowPath(PathPlannerPath, PathConstraints) - Pathfind then follow
